# Contributing to kerb-map

Thanks for your interest in contributing to kerb-map. This guide covers the essentials.

## Getting Started

```bash
git clone https://github.com/b-3llum/kerb-map.git
cd kerb-map
pip install -e .
```

## Project Structure

```
kerb_map/
├── auth/           # LDAP connection & authentication
├── modules/        # Enumeration modules (SPN, AS-REP, delegation, etc.)
│   └── cves/       # Individual CVE check implementations
├── output/         # Reporter (console), exporter (JSON/BloodHound), logger
└── db/             # SQLite scan cache
```

## Adding a New Module

1. Create your module in `kerb_map/modules/` following existing patterns
2. Use `self.ldap.query()` for all LDAP queries (read-only only)
3. Return structured data (dataclasses or dicts)
4. Add a display function in `output/reporter.py`
5. Wire it into `cli.py` (add CLI flag, call module in `run_scan()`, pass to scorer)
6. Add findings to `modules/scorer.py` for attack path ranking

## Adding a CVE Check

1. Subclass `CVECheck` from `modules/cves/cve_base.py`
2. Implement `check()` returning a `CVEResult`
3. Set `noise_level` appropriately ("safe" for LDAP-only, "loud" for RPC)
4. Register in `modules/cve_scanner.py`

## Code Style

- Python 3.10+
- Type hints on public methods
- Use `kerb_map.output.logger.Logger` for console output (not raw `print()`)
- Keep LDAP queries read-only — never modify AD objects

## Safety Rules

- All LDAP queries must be read-only
- RPC probes must be gated behind `--aggressive`
- Document the Windows Event IDs generated by any new check
- Document the noise level in the detection profile

## Submitting Changes

1. Fork the repo and create a feature branch
2. Make your changes
3. Ensure the tool runs without errors on a test domain
4. Submit a pull request with a clear description of what and why

## Reporting Issues

Use [GitHub Issues](https://github.com/b-3llum/kerb-map/issues) for bugs and feature requests.
