# v1.2 known gaps

Honest scope-versus-shipped accounting. Every item below was
identified during the v1.2 development cycle, considered, and
**deliberately deferred** for a documented reason. None are silent
bugs; each names the environment / work needed to close it.

This list is the difference between "code-feature-complete" (which
v1.2.0 is) and "battle-tested across the variety of AD environments
a tool like this needs to handle" (which it isn't yet).

---

## 1. Real-domain validation at scale (only single small domain tested)

**Status:** v1.2.0 was field-validated against ONE Server 2019 lab
domain (~10 users, single child of a 4-domain forest). That single
test surfaced 7 silent-failure field bugs (PRs #29–#35) — extrapolate
to estates kerb-map hasn't seen.

**What's untested:**
- Server 2022 / 2025 forests (BadSuccessor module is the only Server
  2025 surface; not exercised against a real Server 2025 DC)
- 5k+ user estates — paging behaviour, throughput, memory
- Multi-DC sites with referrals
- Hardened estates (signing required, channel binding, restricted RPC)
- Estates with weird schema extensions (Exchange, ConfigMgr, custom)
- Read-only DCs (RODC) — UAC bit 0x4000000 — uncertain if scan adapts

**What it would take:** access to 3–5 different forest topologies
(small / mid / hardened / multi-domain / Server 2025) and an operator
willing to run kerb-map against each + report bugs. Each forest will
likely surface 3–10 bugs the lab didn't, based on the v1.2 ratio.

## 2. Real BloodHound CE 5.x ingest validation

**Status:** kerb-map emits a zip with the four standard JSON files
(users / computers / groups / domains) plus `kerbmap_edges.json`.
The shape was verified against the BH CE 5.x ingest schema docs and
the file structure is correct. **It has not been actually loaded
into a running BH CE instance.**

**What's untested:**
- Whether the `KerbMap*` custom edges render in the UI (they should —
  BH CE accepts arbitrary edge type names — but unverified)
- Cypher queries against the custom edges (e.g.
  `MATCH (u)-[:KerbMapWriteAcl]->(t) RETURN u`)
- Large graphs (50k+ nodes) — ingest time, query latency
- Edge-case node properties that BH CE silently rejects (we may emit
  fields it doesn't recognise; behaviour is undefined)

**What it would take:** stand up a BH CE 5.x docker compose, ingest
a kerb-map scan from the lab, click through the graph, run 3–5
operator-typical Cypher queries, document what works and what
doesn't.

## 4. Coverage on `kerb_map/modules/` ✅ **mostly closed**

Aggregate now at **82.66% on `kerb_map/modules/`** (well above the
brief's 60% target). One straggler:

- `hygiene_auditor.py` — 13% covered, 663 LOC, 280 statements with
  243 missed.

**Why deferred:** hygiene_auditor is essentially 10 sub-modules in a
single file (SID History, LAPS, krbtgt, AdminSDHolder, FGPP,
credential exposure, primary-group abuse, stale computers,
privileged-group breakdown, service-acct hygiene). Each sub-module
has its own LDAP queries, response shape, and risk classifier —
testing all 10 thoroughly is ~80–120 test functions, comparable to
all other modules combined. Out of scope for this PR; better as a
focused follow-up.

**What it would take:** ~3–4 hours of focused test writing. Pattern
is identical to `test_user_enumerator.py` (one section per
sub-method, mock LDAP at the client level).

## 6. Lab provision validation ✅ **closed**

`vagrant up` validated end-to-end on Arch (vagrant 2.4.9 from AUR +
VirtualBox 7.2 + Ubuntu jammy64 base box). Provisioning surfaces 4
real bugs in the lab files which are now fixed:

1. **`ldb-tools` missing** from `provision_dc.sh` apt install. Without
   it, every `ldbmodify` call in the seed silently failed — meaning
   no UAC bit toggling, no key-credential blob, no DCSync ACE seeded.
   kerb-map would then scan a clean lab and find nothing.
2. **`Smith, John` username** rejected by samba (sAMAccountName can't
   contain commas) AND `--surname='Smith, of Sendai'` produced an
   invalid DN with a comma in CN. Dropped — DN-escape testing belongs
   in unit tests, not the lab seed.
3. **`set -e` interaction** with the SID-decoding pipeline (`ldap |
   sed | base64 | python`). On idempotent re-runs the python returned
   empty, the substitution exited non-zero, and `set -e` bailed the
   whole script. Removed `-e` (kept `-uo pipefail`); script is
   designed for partial failure (every line uses `|| true`).
4. **SID decoder Python** raised `IndexError: index out of range` on
   empty objectSid output. Added a length guard.

After fixes: `vagrant provision dc01` runs the full seed end-to-end
and prints `[seed] complete at ...`. Every seeded object verified
present via `vagrant ssh dc01 -c 'samba-tool user list'` /
`samba-tool computer list`.

**One known follow-up surfaced during this validation, NOT closed:**
running kerb-map against the seeded Samba 4 DC fails with
`strongerAuthRequired` on every transport (LDAPS / StartTLS / SASL /
plain). Modern Samba 4 requires LDAP signing for SASL binds, and
ldap3's Kerberos SASL binding doesn't enable signing flags by
default. This is the "hardened estate" gap (#1 below) surfacing in
the lab — kerb-map can scan against the loose Server 2019 in the
culling-game lab but not against signing-required Samba. Fix needs
ldap3 SASL binding to set `sasl_security="sign"` or equivalent.
Tracked as a v1.2.x follow-up.

**Lab is now reproducible** by anyone with vagrant + virtualbox:

```bash
sudo pacman -S vagrant virtualbox  # or apt install equivalent
cd lab && vagrant up
# Provisioning takes ~5–10 min on first run (apt install + samba
# provision + 1500 stub users). On subsequent boots / `vagrant
# provision`, only the idempotent seed runs (~30 s).
```

## 7. Performance baseline (never measured)

**Status:** No throughput / memory / time-to-first-finding numbers
published. Lab DC test (~10 users) completed in 7.8s for the legacy
modules, ~10s with `--v2` — **insufficient signal to extrapolate**.

**What's unmeasured:**
- Scan time on 5k-user estate
- RAM peak on a forest with 50+ GPOs
- LDAP query count per scan (rough estimate: ~20 LDAP queries
  legacy + ~10 per v2 module = ~100 queries per `--all --v2` scan)
- Time-to-first-finding (matters for `--quiet` operators wanting
  early signal)

**What it would take:** access to a 5k+ user lab + `time` wrapping
+ Python's `tracemalloc` / `psutil` instrumentation. Half a day of
benchmarking + `docs/PERFORMANCE.md` write-up.

## 8. Detection profile validation

**Status:** `README.md`'s Detection Profile table is **estimated, not
measured**. Claims like "v2 plugins generate Event 1644 only" or
"Timeroast = NTP request anomaly only" are based on the LDAP query
shape and protocol-level reasoning, **not on watching a SOC analyst's
SIEM during a kerb-map run**.

**What's unmeasured:**
- Actual Win Event IDs generated per module
- MDI alert classes triggered
- EDR signatures hit
- Specific 4769 / 4625 / 5145 counts per scan

**What it would take:** sit alongside a SOC analyst with Splunk /
Sentinel / MDI open during a kerb-map run. A 30-minute coordinated
run would produce enough signal to update the table with measured
values + IDs.

## 9. GPP SMB-grep plumbing

**Status:** PR #30 was an honest fix — GPP no longer false-flags
HIGH on every clean domain. But the check still doesn't actually
verify `cpassword=` in SYSVOL XMLs; it reports INDETERMINATE and
asks the operator to grep manually.

**What it would take:** plumb operator credentials past the LDAP
layer into the CVE infrastructure (`CVEBase` currently has only
`ldap`, `dc_ip`, `domain` — not user/pass). Then `gpp_passwords.py`
could open an SMB session via impacket's `SMBConnection`, walk
`SYSVOL\<domain>\Policies\*\*\*.xml`, and grep. ~150 LOC + tests.

This is a meaningful improvement (GPP is one of the highest-impact
findings on legacy estates) but cuts across the v2 module
infrastructure and deserves a focused PR after v1.2.0 ships.

---

## 10. Operator iteration loop (not run yet)

**Status:** v1.2.0 is shipping with a single round of field
validation by one operator (the development session). Real
engagements surface dozens of rough edges per week — confusing
error messages, unexpected ACE shapes, weird LDAP referral chains,
AD bugs that crash impacket, environments where Python 3.x has a
quirk. **None of that has happened yet.**

**What it would take:** ship v1.2.0 to 3–5 trusted operators, gather
issues over 2–4 weeks, triage and patch. The bug volume should taper
as edge cases get fixed; expect another 5–10 PRs of the
PR-#29-through-#35 calibre.

## Roadmap

A reasonable v1.2.x → v1.3 trajectory closes these in roughly this
order:

1. **v1.2.1** — operator iteration patches (#10), dMSA / schema-
   tolerance pattern extended to other Server 2025-only classes
   (msDS-AuthN-PolicySilo, etc.).
2. **v1.2.2** — hygiene_auditor coverage (#4 stragglers) + GPP
   SMB-grep (#9).
3. **v1.3.0** — kerb-chain integration (#5) + cross-domain trust
   walking + BloodHound CE ingest validated against real instance
   (#2) + performance baseline (#7).
4. Out-of-band: lab provision validation (#6) and detection profile
   measurement (#8) — both depend on hardware / coordination, not
   code.
